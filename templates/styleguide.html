    CSS: Setup

    The project uses Sass to provide the CSS with some structure.

    • You need to install Ruby and Sass.
    • Watching and compiling can be done by running one single command, preferably outside the vagrant box: 
      sass --watch assets/static/sass/style.scss:assets/static/css/style.css.
    • Don't edit the generated style.css in /assets/static/css
    CSS: Structure
    • The partials folder inside the sass folder contains all the small Sass files that are imported into the style.scss folder
    • The media-queries partial contains all the styles that make the app responsive.
    • Reference for typography, e.g. headings, links, lists belong to the _typography.scss partial. These affect the entire app; handle with care if making changes.
    • Reference for forms: e.g. inputs, labels, selects, errors is found in the forms partials. All overrides for individual forms are made here.
    • Reference for globals: vendor prefixing mixins and global classes belong to the _globals.scss partial. These affect the entire app; handle with care if making changes.
    • All variables referenced in all other Sass files belong to the variables partial. A variable is always prefixed with a $ symbol. e.g. $header-height defines the height of the fixed height header, which can then be re-used for positioning the main element, map, fixed alerts, etc.
    • The lib folder inside the partials folder, contains all third party plugin sass partials
    • If you need a new UI module that deserves its own partial file, e.g. some groovy animation library, you could create a new partial (e.g. _groovy.animation.scss) and import it into the style.scss file. Documentation here.
    CSS: Conventions
  • Do use 2 spaces to indent.
  • Don't use ids for styling; they are used for JS
  • You can use variables, nesting, mixins, extends and placeholders in a manner that benefits the app. Some documentation if you're new to Sass.
  • Do use Sass comments (similar to JS single line comments //) for comments that only other developers need to see. These do not get compiled into the final CSS stylesheet.
  • Do use helpful comments for new sections inside a Sass partial, which are useful for documentation of the section in question. See below: 
              
/*-------------------------------------*/
/* FORM: BASE STYLES */
  • Do spend a few minutes browsing through the globals, variables and typography partials. They contain plenty of reusable code.
  • Don't try to extend a Sass placeholder (e.g. %grid-parent) inside a media query, because you will get an error and the file won't compile. If a mixin is available, use that instead, inside the media query block.

    Grid: Media Queries

    By and large, the site uses min-width media queries, on the assumption that the initial layout is mobile-first and stacked, and columns are added incrementally, as the screen size gets larger.

    • The 3 media min-width queries are 600px, 750px and 990px
    • The 2 max-width media queries are 599px and 749px. These are used sparingly.

    Grid: Containers

    • The site has two container sizes you can use.
    • .container, which is the widest the grid can get i.e. 1000px
    • .container-sm, which is a smaller container, used for convenience i.e. 520px

    Grid: Classes

    There's no real grid, in case you were wondering. But there's enough stuff to build flexibly.

    You can add more classes as required in the grid partial; the current grid simply accommodates 1/2, 1/3rd, 2/3rds and 1/4th widths.

    Building Blocks:

    • Firstly, add a section with a container class. (This is optional if a parent container class already exists in the DOM).
    • Next, add a parent for your grid: with a class of grid-parent (this internally has a clearfix, so you don't need to worry about clearing floats).
    • Add some children with a class of grid-child, to get some sweet grid gutters.

    Mixins instead of grid classes

    The demos above will work for sections that don't need to be stacked or have their grids changed on different media breakpoints.

    Since the site is built mobile first, it would be preferable not to add classes, as indicated in the demos above, since they would have to be reset for mobile, to become full-width sections (in most cases, but not all).

    Instead, consider using the handy grid mixins available on media query breakpoints. The mixins have the same names as the grid classes, i.e. grid-parent, grid-child, col-25-flt, col-50-flt, col-33-flt, col-66-flt

    Let's look at a simple demo below, using classes from the existing site. On a media query breakpoint of 750px, the parent class partners-list-block includes the mixin grid-parent, and the children partners-list include the mixin col-50-flt.

    Thus you can float columns of any widths (that add up to 100%) on any media query, to get a truly flexible grid.
